# Lessons

## 2026-03-20

- 当数据库 schema 变更需要兼容“已有库”时，不能只更新 `scripts/init_db.sql` 或依赖运行时自建 schema；必须同时提供单独的增量迁移 SQL。
- 用户指出交付物缺口后，要把缺口转成回归测试，避免再次出现“功能实现了，但运维落地件缺失”的问题。
- 新增“自动选择器”这类基础能力时，要优先做启发式短路和懒加载依赖，避免每个请求都初始化真实 LLM 客户端，尤其在测试和受限环境里。
- 当同一产品同时提供一方 API 和 OpenAI 兼容层时，涉及“转人工 / 客服 / 会话接管”这类用户可见能力，不能只在一方 API 接线；兼容层也要做等价分流或明确拒绝，并用回归测试锁住。
- 新增与 agent、skill、planning、memory、handoff、tool orchestration 相关的能力时，先核对 Deep Agents 官方是否已有标准实现；默认优先采用官方机制，只有明确不满足需求时才允许自定义，并在计划里写清差距与原因。
- 使用 Deep Agents 官方 `skills=[...]` 时，不能依赖 `FilesystemBackend` 默认配置；必须显式评估 backend 根目录和 `virtual_mode`，否则文件工具可能暴露超出预期的路径范围。
- 当官方 skill 机制无法直接覆盖现有非 deepagents 子智能体时，不要继续维护平行的硬编码 prompt；应复用同一份 `src/skills` 资产作为单一事实源，并用共享 loader 保持行为一致。
- 删除"死代码守卫"（如 `AgentExecutor = None`）时，必须同步更新对应测试中的 `patch` 调用；否则测试会因 `AttributeError: module has no attribute` 而崩溃，即使测试本身并不依赖该守卫路径。
- 在 Python 文件中间位置放置 `import` 语句（模块级别的 asyncio.Lock 初始化）会触发 E402；应在文件顶部统一 import，在模块末尾使用已导入的名称初始化单例锁。
- 外部输入触发的 HTTP 响应不应携带 `str(e)` 异常详情；应在 logger 记录完整错误，对客户端只返回通用提示，防止内部调用栈和数据库路径泄漏。
- 对 webhook、router、startup path 这类导入即执行的模块，外部可选依赖不要做模块级强导入；优先改成懒加载或显式降级，否则环境里少一个包就会让整个服务无法启动。

## 2026-03-23

- 当用户明确要求“只支持新会话、不兼容旧会话”时，不要默认保留回退路径；接口、列表过滤、详情读取和测试都要同步切到新边界，否则会把旧模型意外带回实现里。
- 当用户要求“SQL 相关操作改用 SQLAlchemy 封装”时，不能只替换新增功能里的查询；应先盘点所有自写存储层，再统一切到底层共享 engine/table metadata，否则会留下 `asyncpg`/SQLAlchemy 双栈并行的维护负担。

## 2026-03-24

- 共享数据库访问层已经统一到 SQLAlchemy 时，不要让同一服务在 `psycopg` 和 `asyncpg` 之间分裂驱动；尤其请求取消和流式断开场景下，驱动差异会直接表现为不同的连接回收行为和噪音异常日志。
- 在 SQLAlchemy Core/AsyncEngine 里，`engine.connect()` 适合读，不适合执行需要持久化的 `INSERT/UPDATE ... RETURNING`；这类写语句必须包在 `engine.begin()` 或显式提交事务里，否则“拿到了返回行”也会在连接关闭时被回滚。
- 当用户把“空字符串应视为新会话”说得更具体时，不要停留在代码里已经覆盖 `""` 的表象；要继续检查纯空白字符串、schema 层归一化和回归测试覆盖，否则同一需求会以 `"   "` 这种边界值重新漏出来。

## 2026-03-27

- 写 shell 脚本时，如果目标环境可能落在 macOS 自带 Bash 3.2，就不要假设 `set -u` 下空数组展开和新版本 Bash 一样安全；对可选数组参数必须先判长度，再展开，并用真实 `bash script.sh ...` 回归测试覆盖。

## 2026-03-30

- Docker bind mount 会覆盖镜像构建期对目录做的 `chown` 结果；只要容器以固定非 root UID 运行，涉及宿主机挂载写目录时，就必须在部署脚本/文档里把宿主机目录 owner 对齐到该 UID，并在启动时做一次真实写探针，不能只 `mkdir(exist_ok=True)`。
- 生产部署脚本只要用了 `rsync --delete`，就必须先盘点远端目标目录下是否混放持久化卷目录；像 `postgres/`、`etcd/`、`minio/`、`milvus/`、`backups/` 这类运行时数据目录必须显式排除，不能只排除 `uploads/` 这种单一路径。
- 对 MinIO、Milvus 这类状态型依赖，`docker healthcheck` 只证明进程活着不够；部署后还要补一层存储元数据/最近错误日志校验，否则像 `listPathRaw: 0 drives provided` 这种底层卷异常会在“容器 healthy”状态下漏过去。
- 对云存储、XML 解析器这类可选或环境敏感依赖，不要在服务启动主链路上做模块级强导入；优先懒加载，让缺依赖时只影响对应功能，而不是让 `src.main` 整体导入失败。
- 外部存储配置出错时，不要只盯着网络超时表象；先把运行中的目标 host 从日志里抠出来对照 `.env`，像 `allyuncs.com` 这类拼写错误会伪装成超时。与此同时，接口层必须把 OSS/对象存储异常转换成明确的 5xx，而不是把 SDK 栈直接漏给用户。
- 当附件内容的事实源已经切到 OSS 时，不要再让 `/chat`、`/chat/stream`、OpenAI 兼容层各自落不同的本地临时文件路径；应统一收口到 `UPLOAD_DIR` 下的受控缓存目录，否则会和下游工具的路径白名单、权限模型打架。
- 附件解析层要对空附件列表先短路；否则即使请求没有附件，也会平白初始化数据库连接，把环境缺依赖问题放大成所有聊天请求都失败。
- 当沙箱代码执行器在 AST 层允许 `import pandas as pd` 这类白名单导入时，运行时也必须提供受控的 `__import__`；否则模型会持续生成“看起来合法、实际上必失败”的代码，在 agent tool loop 里自旋到递归上限。
- 允许 `import pandas` 不等于放开真实 pandas 文件访问；如果系统依赖 `UPLOAD_DIR` 做本地文件白名单，就要把导入到的 `pandas` 映射到带路径保护的 safe wrapper，而不是直接暴露原始模块。
